Build Flashpart like an operations tool, not a generic catalog manager. The goal is to make repetitive physical work fast, calm, and hard to mess up: every screen should reduce uncertainty, preserve momentum, and make the next right action obvious.

Prefer durable workflow state over clever UI state. Automation should be traceable and reversible where it matters, especially when it crosses into real-world inventory, photos, or Shopify. Keep the interface boring in the best way: familiar controls, clear statuses, and no hidden magic.

When choosing between speed and trust, protect trust. Move quickly by using proven building blocks, but make the system explicit about what it knows, what is pending, and what needs human review.

## Desktop / mobile split

The UI has one router and one set of pages, but two shells. The split happens in exactly one place: `src/components/shell/app-shell.tsx` chooses `DesktopShell` (top nav, ops console) or `MobileShell` (bottom tabs, capture-first) via `useIsMobile()` from `src/lib/use-is-mobile.ts`. Never branch on user agent or window width anywhere else; pages adapt with responsive Tailwind classes, or `useIsMobile()` when truly necessary. Mobile is an installable iOS-first PWA: icons are generated by `scripts/generate-icons.mjs`, haptics come from `src/lib/haptics.ts` (hidden-switch trick), and capture routes render full-screen without shell chrome.
